Person 1: Agent Brain / Conversation / CPR Logic
Person 1 builds what the AI says, how it decides things, and how it reacts.
1. Define agent scope and behavior
Person 1 should:
define that the agent only handles suspected cardiac arrest


define what the agent is allowed to do:


ask assessment questions


guide CPR


react to XR feedback


answer CPR-related questions


define what the agent is not allowed to do:


diagnose medical conditions


give medication advice


go outside CPR scope


define the tone:


calm


short


direct



2. Build the state machine logic
Person 1 should:
use the shared state names exactly:


SESSION_START


RESPONSIVENESS_CHECK


BREATHING_CHECK


CPR_INSTRUCTIONS


CPR_ACTIVE


REASSESSMENT


WAIT_FOR_EMS


SESSION_END


define the rules for moving between states


decide when CPR starts


decide when CPR stops


hardcode critical transitions so the AI does not improvise


Example:
if person is unresponsive and not breathing normally → move to CPR_INSTRUCTIONS


when CPR begins → move to CPR_ACTIVE


if breathing resumes → move to WAIT_FOR_EMS



3. Build the intake question flow
Person 1 should:
write the exact first lines the agent says


write the exact questions in order


keep the flow short and action-oriented


make sure the agent asks one question at a time


handle answers like:


yes


no


not sure


define follow-up wording for unclear responses


decide what the agent says after each answer


Typical questions:
“Is the person awake or responding?”


“Are they breathing normally?”



4. Build the condition evaluation logic
Person 1 should:
read responsiveness_status


read breathing_status


decide whether the agent should:


start CPR


continue checking


move to monitoring


safely handle uncertain answers


make sure the agent never gets stuck


This is the logic layer that turns answers into action.

5. Build the CPR instruction engine
Person 1 should:
write the CPR instruction sequence


split it into small spoken steps


define what the agent says in CPR_INSTRUCTIONS


define what the agent says in CPR_ACTIVE


Examples of content:
where to place hands


how to position arms


push hard and fast


follow the rhythm


keep going



6. Build the XR reaction logic
Person 1 should:
decide how the agent reacts to XR signals


use the exact shared XR signal names:


hand_position


compression_rhythm


use the exact shared values:


correct


incorrect


not_detected


slow


good


fast


Mapping examples:
if hand_position.status = incorrect → tell user to reposition hands


if rhythm status = slow → tell user to go faster


if rhythm status = fast → tell user to slow down


if rhythm status = good → encourage user to continue



7. Build the CPR coaching loop
Person 1 should:
make the agent continue coaching while current_state = CPR_ACTIVE


decide how often the agent speaks


avoid speaking too much


define encouragement phrases


define correction phrases


allow the loop to react to live XR feedback


allow the loop to stop if condition changes



8. Build the reassessment logic
Person 1 should:
define when to move to REASSESSMENT


write the reassessment prompts


decide what happens if the person starts breathing again


decide what happens if nothing changes


move to WAIT_FOR_EMS when appropriate



9. Build the RAG answer behavior
Person 1 should:
decide when the agent should answer a CPR question


decide when it should ignore extra questions and keep focusing on CPR


make retrieval answers short and useful


keep the scope narrow


Examples of allowed questions:
where to put hands


what if they are gasping


when to stop



10. Build the system prompt and safety behavior
Person 1 should:
write the system prompt


force the model to stay in scope


force concise responses


force action-first behavior


prevent diagnosis claims


prevent unsafe medical advice


define what the model says if asked something outside scope



11. Connect to shared tools
Person 1 should call these exact tool names when needed:
get_location


log_event


update_incident_status


record_user_response


generate_incident_summary


Person 1 should not rename them.

12. Person 1 deliverables
Person 1 should produce:
conversation flow


state transition logic


CPR scripts


reassessment scripts


XR signal reaction rules


system prompt


agent orchestration code
